## 2. Failure-Path Reconstruction

**Normal path**: Review fixes should leave each workflow artifact and skill instruction aligned with the current contract: unique YAML keys, canonical status/auxiliary vocabulary, and one template for generated artifact frontmatter.

**Failure path**: During status/control-plane canonicalization, wording was updated in many places, but four small inconsistencies survived: a duplicate key in a checklist, a stale enum value in onboard prose, a stale hyphenated field name in explore prose, and an architecture frontmatter field required only in prose.

**Split point**: the changed files above — each local inconsistency is enough for future agents or validators to write divergent artifacts.

## 3. Root Cause

**Root-cause type**: data-format / contract drift.

**Root-cause description**: PR #2 changed the workflow artifact vocabulary from older lifecycle terms into canonical fields, but the final static review did not include duplicate-key detection and did not catch all prose/template alignment points. The result is not an implementation algorithm bug; it is schema/prose drift in workflow-contract documents.

**Are there multiple root causes?**: yes, four local causes under the same class:

1. Duplicate YAML mapping key in the workflow-control-plane checklist.
2. Onboard prose uses an enum value not present in the config contract.
3. Explore prose uses the old `superseded-by` spelling.
4. Architecture prose and architecture template disagree about `current: true`.

**Complex bug diagnosis record**: not triggered. The failure is stable, static, and each location is directly visible in the changed text.

- **feedback loop**: static file inspection plus targeted grep/validation.
- **ranked hypotheses**: not needed; root causes are directly visible.
- **instrumentation / measurement**: none.

- **impact scope**: limited to PR #2 workflow-contract artifacts and skill instructions.

## 5. Repair Options

### Option A: Minimal direct contract repair

- **what it does**: remove the duplicate `checks:` key; change `not_configured` to the already-allowed `unavailable`; change `superseded-by` to `superseded_by`; remove the prose-only `current: true` requirement from `bt-arch` backfill wording so it matches the existing architecture template.
- **advantages**: smallest change, no new schema value, no new architecture frontmatter field, directly addresses all four review findings.
- **disadvantages / risks**: architecture docs will rely on the existing convention that architecture records current state by definition, rather than an explicit `current: true` field.
- **impact surface**: four files changed; no runtime behavior.

### Option B: Expand schemas to include every prose value

- **what it does**: add `not_configured` to both config references and add `current: true` to the architecture reference template, plus fix the duplicate checklist key and superseded field spelling.
- **advantages**: preserves the existing prose where possible.
- **disadvantages / risks**: expands the machine-readable vocabulary and architecture frontmatter without a need; more future surface area.
- **impact surface**: six files changed; more schema churn.

### Option C: Add a validator enhancement first

- **what it does**: update validation tooling to detect duplicate keys and maybe stale auxiliary field spellings, then fix the files.
- **advantages**: prevents some future misses.
- **disadvantages / risks**: larger scope than the reported PR review findings; tool behavior is outside the direct fix.
- **impact surface**: script plus docs; unnecessary for this issue.

### Recommended Option

**Recommend option A**, because it is the smallest repair, uses existing canonical vocabulary, and avoids expanding schemas just to accommodate stale prose. The user's instruction “走issue流程修复上面所有4个问题” is treated as confirmation to apply the recommended minimal repair.
